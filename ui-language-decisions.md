# View Display Language Decision Sheet

View の表示言語を日本語 / English で統一的に切り替えるために、実装前に決めておくべき項目を表にまとめたものです。

使い方:

- 「回答」列に採用する方針を書いてください。
- 「候補 / 判断材料」はそのまま残しても、不要なら削っても構いません。
- 未決定の項目は空欄のままで進捗管理に使えます。

| No. | 決定事項                     | 何を決めるか                                                                   | 候補 / 判断材料                                                                                                                         | 回答 |
| --- | ---------------------------- | ------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------- | ---- |
| 1   | 切替対象の範囲               | どこまでを「表示言語切替」の対象に含めるか                                     | 例: メニュー / ボタン / タブ名 / プレースホルダー / ダイアログ題名 / ステータスバー / 確認ダイアログ / エラーメッセージ / HTML 初期文言 |      |
| 2   | UI表示言語とAI出力言語の分離 | UI の言語設定と Gemini 応答の output_language を別設定にするか                 | 推奨: 別設定にする。UI 表示と AI 応答の責務が異なるため                                                                                 |      |
| 3   | 設定名                       | UI 表示言語を設定として何という名前で持つか                                    | 例: ui_language / display_language / locale                                                                                             |      |
| 4   | 管理場所                     | 表示言語の保持元をどこに置くか                                                 | 例: AppConfig に永続化 / 一時状態のみ / Qt のロケール依存                                                                               |      |
| 5   | 初期値                       | 初回起動時の言語をどう決めるか                                                 | 例: 常に日本語 / OS ロケールで自動判定 / 英語固定                                                                                       |      |
| 6   | 永続化方針                   | 次回起動時に前回の選択を復元するか                                             | 推奨: AppConfig に保存して復元                                                                                                          |      |
| 7   | 翻訳方式                     | 文言管理の仕組みをどうするか                                                   | 例: Qt QTranslator / 独自辞書 + translation service / View ごとの定数辞書                                                               |      |
| 8   | 文言キー設計                 | 文言をキーで管理するか、画面単位の辞書で管理するか                             | 例: main.menu.file / settings.tab.ai_models のような階層キー                                                                            |      |
| 9   | Presenter 文言の扱い         | Presenter が生成しているユーザー向け文字列も切替対象にするか                   | 現状は status / confirm / error 文言が Presenter にも存在するため、統一切替なら対象に含めるのが自然                                     |      |
| 10  | View の責務                  | 翻訳は View 側で行うか、Presenter 側で解決済み文字列を渡すか                   | Passive View を維持するなら、翻訳サービスを Presenter でも使える設計が必要                                                              |      |
| 11  | 即時反映か再起動反映か       | 設定変更後にその場で UI を切り替えるか                                         | 例: 即時反映 / アプリ再起動時反映 / ダイアログのみ次回表示から反映                                                                      |      |
| 12  | 反映対象ウィンドウ           | すでに開いている画面をどう更新するか                                           | 例: MainWindow と SidePanel は即時更新、SettingsDialog / CacheDialog は次回生成時に新言語適用                                           |      |
| 13  | 設定UIの配置                 | 表示言語の設定項目をどこに置くか                                               | 例: General タブ新設 / Rendering タブに追加 / 既存タブを再編成                                                                          |      |
| 14  | 言語選択 UI                  | ユーザーに何を見せるか                                                         | 例: 日本語 / English 表記、または Japanese / English 表記、あるいは両方併記                                                             |      |
| 15  | メニューアクセラレータ       | 言語ごとにメニューの (&F) のようなアクセラレータをどうするか                   | 例: 各言語で個別管理 / 英語のみ付与 / 日本語でも維持                                                                                    |      |
| 16  | ショートカット表記           | メニュー名変更時もショートカット説明をどう見せるか                             | 例: 文言だけ翻訳し、ショートカットは共通のまま                                                                                          |      |
| 17  | 単位と書式                   | TTL・残り時間・tokens・ON/OFF・Active/Inactive などの表記をどう統一するか      | 例: 英語では min / expired、日本語では 分 / 期限切れ                                                                                    |      |
| 18  | 空状態の表記                 | モデル未設定、データなし、結果未表示などの空状態文言をどうするか               | 例: --- / None / Not set / 未設定 を一元管理する                                                                                        |      |
| 19  | エラー文言の方針             | 例外由来の生メッセージをそのまま見せるか、前置きだけ翻訳するか                 | 例: UI だけ翻訳して詳細は原文維持 / 完全翻訳しない / ログと UI を分離                                                                   |      |
| 20  | HTML 初期文言の扱い          | QWebEngineView に入れている初期 HTML メッセージを切替対象に含めるか            | 現状は「翻訳結果がここに表示されます」「結果がここに表示されます」が埋め込み文字列                                                      |      |
| 21  | ダイアログ同期APIの文言      | show_error_dialog / show_confirm_dialog の title と message を誰が組み立てるか | 例: 呼び出し側で両方翻訳済み文字列を渡す / View 側でキーから生成する                                                                    |      |
| 22  | 未翻訳キーのフォールバック   | 辞書に存在しないキーの扱いをどうするか                                         | 例: 日本語にフォールバック / 英語にフォールバック / キー名を表示 / 開発時のみ例外                                                       |      |
| 23  | ロケール表現                 | 内部値を何で持つか                                                             | 例: ja / en / ja-JP / en-US                                                                                                             |      |
| 24  | 対応言語の拡張性             | 今回は 2 言語だが、将来の 3 言語目以降を考慮するか                             | 例: 今回から拡張可能な辞書構造にする / まず ja-en 固定で実装する                                                                        |      |
| 25  | テスト範囲                   | 何をもって実装完了とするか                                                     | 例: View の静的文言だけ / Presenter 発メッセージも含む / 切替後の即時反映まで含む                                                       |      |
| 26  | スクリーンショット差分確認   | UI 確認をどの粒度で行うか                                                      | 例: MainWindow / SettingsDialog / CacheDialog / SidePanel の両言語を確認                                                                |      |
| 27  | 既存設定ファイル互換         | ui_language を追加したとき旧 config.json をどう扱うか                          | 現状の load_config は未知フィールドを無視し既知フィールドのみ読むので、追加互換は取りやすい                                             |      |
| 28  | ドキュメント更新             | README や仕様書に表示言語設定を追記するか                                      | 例: README の設定項目に追加 / 実装後にまとめて更新                                                                                      |      |

## 推奨の初期方針メモ

未決定なら、最初のたたき台としては次の方針が扱いやすいです。

| 項目                   | 推奨案                                                          |
| ---------------------- | --------------------------------------------------------------- |
| UI表示言語とAI出力言語 | 分離する                                                        |
| 管理場所               | AppConfig に ui_language を追加して保存する                     |
| 翻訳方式               | 独自の小さな翻訳辞書 + translation service                      |
| 反映タイミング         | MainWindow / SidePanel は即時反映、ダイアログは次回生成時に反映 |
| 切替対象               | View の静的文言 + Presenter のユーザー向けメッセージ全体        |
| 初期値                 | OS ロケールが ja 系なら日本語、それ以外は English               |
